    # RSS가 없는 사이트를 강제로 크롤링하여 저작권 침해를 방지하는 방법
    
    본 프로젝트는 공식적으로 RSS를 제공하는 언론사의 RSS 피드만을 수집 대상으로 사용하도록 설계하였다.
    현재 수집 대상은 다음과 같다.
    
    • 연합뉴스 RSS
    • 한국경제 RSS
    • 파이낸셜뉴스 RSS
    
    각 언론사의 공식 RSS URL을 HTTP Request 노드에서 직접 호출하여 XML 데이터를 수집한 후,
    XML to JSON 변환 및 Normalize 과정을 거쳐 기사의 제목, 링크, 발행일, 언론사 등의 메타데이터만 추출한다.
    따라서 RSS를 제공하지 않는 사이트를 임의로 탐색하거나 HTML을 분석하여
    기사 목록을 생성하는 방식의 크롤링은 수행하지 않는다.
    
    ## Top3 기사 검증 방식
    뉴스 선정 이후에는 AI가 선택한 Top3 기사에 한하여 기사 URL에 접근한다.
    이 과정의 목적은 새로운 기사를 수집하기 위한 것이 아니라,
    
    • 기사 URL 정상 여부 확인
    • 기사 메타데이터 검증
    • 저작권 관련 정보 확인
    
    등을 수행하기 위한 절차이다.
    
    즉, 기사 원문을 수집하거나 저장하는 것이 아니라,
    URL에 접근하여 필요한 정보를 검증하는 과정이다.
    
    검증이 완료되면 HTML 데이터는 별도로 저장하지 않으며,
    최종적으로 Notion 데이터베이스에는 다음 정보만 저장된다.
    
    • 기사 제목
    • 기사 링크
    • 언론사
    • 발행일
    • AI가 생성한 요약
    
    기사 원문이나 HTML 데이터는 저장하거나 재배포하지 않으며, 데이터베이스에도 기록되지 않는다.
    
    ## 저작권 침해 방지를 위한 설계
    본 워크플로우는 저작권 침해 가능성을 최소화하기 위해 다음과 같은 원칙을 적용하였다.
    
    • 공식 RSS를 제공하는 언론사만 수집 대상으로 사용
    • RSS가 없는 사이트는 크롤링 대상에서 제외
    • 기사 원문은 저장하거나 재배포하지 않음
    • 기사 메타데이터와 AI 요약만 저장
    • Top3 기사에 대해서만 URL 접근을 통해 메타데이터와 저작권 정보를 검증
    
    ## 기대 효과
    이와 같은 구조를 통해
    
    • 비공식 웹 크롤링 방지
    • RSS 미제공 사이트의 무단 수집 방지
    • 기사 원문 재배포 방지
    • 저작권 침해 가능성 최소화
    
    를 달성하도록 설계하였다.

# **▶️ 주제 필터링 기준 및 Top3 선정 기준**
written by 김태영

## 1. 주제 필터링(Topic Filter)

    뉴스 자동화 워크플로우에서는 RSS를 통해 수집된 모든 기사를 바로 AI에게 전달하지 않고,
    먼저 Topic Filter를 통해 원하는 주제에 해당하는 기사만 선별하도록 설계하였다.
    
    주제는 경제, IT, 국제, 정치, 생활의 다섯 가지로 구성하였으며, 각 주제마다 관련성이 높은 핵심 키워드를 정의하였다.
    • 경제 : 경제, 주식, 증시, 금리, 환율, 투자, 부동산
    • IT : AI, 반도체, 로봇, 배터리, 클라우드, 데이터센터
    • 국제 : 미국, 중국, 일본, 유럽, 중동, 러시아
    • 정치 : 대통령, 국회, 정부, 선거
    • 생활 : 교육, 복지, 교통, 의료, 물가
    
    Topic Filter는 기사의 제목(Title) 과 설명(Description) 에 포함된 키워드를 기준으로 관련 여부를 판단하며,
    선택된 주제와 관련된 기사만 다음 단계로 전달한다.
    이와 같은 구조를 적용한 이유는 모든 기사를 AI가 분석하도록 할 경우 처리 비용이 증가하고,
    주제와 무관한 기사까지 포함되어 분석 정확도가 낮아질 수 있기 때문이다.
    따라서 1차적으로 주제와 관련된 기사만 선별하여 AI에 전달함으로써 다음과 같은 효과를 기대하였다.
    • AI 분석 대상 축소
    • 처리 비용 절감
    • 주제별 뉴스 분류
    • Top3 선정 정확도 향상
    
## 2. Top3 뉴스 선정 기준
    Topic Filter를 통과한 기사들은 AI가 중요도를 분석하여 최종적으로 오늘의 주요 뉴스 3건을 선정하도록 구성하였다.
    Top3 선정 시 적용한 기준은 다음과 같다.
    • 국민 생활에 미치는 영향
    • 경제에 미치는 영향
    • 국제 정세에 미치는 영향
    • 장기적인 사회적 파급력
    • 동일 사건의 중복 기사 제거
    • 정치적 편향 없이 객관적인 기준 적용
    
    특히 단순히 조회수가 높은 기사나 화제성이 높은 기사보다, 
    국민 생활과 사회에 실질적인 영향을 주는 뉴스를 우선적으로 선정하도록 프롬프트를 설계하였다.
    또한 동일한 사건을 여러 언론사가 반복 보도하는 경우에는
    하나의 기사만 선택하도록 하여 다양한 이슈가 선정될 수 있도록 하였다.
    이러한 기준을 적용함으로써 뉴스 자동화 시스템은 단순한 기사 수집이 아니라,
    중요도를 고려한 뉴스 큐레이션 서비스의 역할을 수행하도록 설계하였다.

## 1. 현재 중복 방지 구조

현재 워크플로우에는 이미 2단계로 중복 방지가 구성되어 있습니다:

1. **Merge 노드**: `"numberInputs": 3`으로 설정되어 3개 소스의 기사를 단순히 **Append(합치기)** 합니다.
2. **Deduplicate News 노드**: JavaScript Code 노드에서 `link`와 정규화된 `title`(앞 35자)을 기준으로 중복을 제거합니다.

### **개선 방안**

#### **방법 1: Merge 노드 자체에서 중복 제거 (권장하지 않음)**

n8n Merge 노드 v3.x의 **"Combine" → "Remove Duplicates"** 모드는 **2개 입력** 간의 비교에 최적화되어 있어, 3개 입력을 동시에 비교하기엔 적합하지 않습니다. 따라서 현재처럼 **Append 후 별도 중복제거**가 올바른 패턴입니다.

#### **방법 2: n8n 내장 "Remove Duplicates" 노드 사용 (간편한 대안)**

커스텀 JavaScript Code 대신 n8n 내장 **Remove Duplicates** 노드(`n8n-nodes-base.removeDuplicates`)를 사용할 수 있습니다:

- **Deduplicate By**: `link` 필드 기준
- 설정이 간단하고 유지보수가 쉬움

#### **방법 3: 현재 Code 노드 개선 (현재 가장 강력)**

현재 **Deduplicate News** 코드 노드는 이미 **이중 검증**(link + title 35자)을 하고 있어 가장 강력한 방식입니다. 다만 개선할 점이 있다면:

| **현재 로직** | **개선 가능 사항** |
| --- | --- |
| `link` 기준 중복 제거 | URL 쿼리 파라미터 제거 후 비교 (`?utm_source=...` 등) |
| `title` 앞 35자 비교 | 더 긴 범위(예: 50자) 또는 유사도 비교 |
| 최대 80건 반환 | 필요에 따라 조정 |

## 2. 요약 호출 기사 1건당 재시도는 최대 2회까지만 가능하도록 설정

현재 `AI 요약` 노드(**L200-L236**)와 `Top3 선정` 노드 모두에 **재시도 설정이 전혀 되어 있지 않습니다.**

n8n에서 재시도를 활성화하려면 노드에 아래 필드가 있어야 합니다:

```
json
"retryOnFail":true,
"maxTries":3,// 최초 1회 + 재시도 2회
"waitBetweenTries":1000
```

현재는 이 필드들이 모두 빠져 있어, API 호출 실패 시 **바로 에러가 나고 워크플로우가 중단**됩니다.


### **재시도 설정 추가 방법**

`AI 요약` 노드에 최대 2회 재시도를 추가하려면 두 가지 방법이 있습니다:

**방법 A: n8n UI에서 설정 (추천)**

- 해당 노드 클릭 → **Settings 탭** → **"On Error"** 섹션에서 `Retry on fail` 활성화 → Max tries: `3` (최초 1 + 재시도 2)

**방법 B: JSON 직접 수정** `AI 요약` 노드의 `"credentials"` 바로 아래에 다음을 추가하면 됩니다:  (수정완료)

```
  "id": "0fbe379b-af2a-474a-ae48-4025e64385f8",
  "name": "AI 요약",
  "retryOnFail": true,
  "maxTries": 3,`
  "waitBetweenTries": 1000,
  "credentials": { "openAiApi": { "id": "8qXymgPOnAZUggvG",`
```

## 3. 무한 루프에 빠지지않도록 트리거 설정 주의 적용

현재 워크플로우는 **구조적 무한루프**(A→B→A 같은 순환 연결)는 없습니다. 일직선 흐름이에요. 하지만 **무한대기/정지 없는 실행** 위험은 존재합니다:

### **🔴 발견된 위험 요소**

| **#** | **위험** | **해당 노드** | **현재 상태** |
| --- | --- | --- | --- |
| 1 | **워크플로우 전체 타임아웃 없음** | `settings` | `executionTimeout` 미설정 → 무제한 실행 가능 |
| 2 | **HTTP Request 타임아웃 없음** | YNA, Hankyung, fnnews | RSS 서버가 응답 안 하면 무한 대기 |
| 3 | **OpenAI 호출 타임아웃 없음** | Top3 선정, AI 요약 | API 응답 지연 시 무한 대기 |
| 4 | **에러 처리 정책 없음** | 전체 노드 | `onError` 미설정 → 하나라도 실패하면 전체 중단 |
| 5 | **빈 결과 가드 없음** | Merge → 이후 노드들 | 3개 소스 모두 실패 시 빈 데이터로 AI 호출 → 토큰 낭비 |

### **🟢 권장 방지 장치**

**① 워크플로우 전체 실행 타임아웃** (가장 중요)

```
json
"settings": {
"executionTimeout":300,// 5분 초과 시 강제 종료
"executionOrder":"v1",
"binaryMode":"separate"
}
```

**② HTTP Request 노드에 타임아웃 추가** (YNA, Hankyung, fnnews)

```
json
"options": {
"timeout":10000// 10초 초과 시 실패 처리
}
```

**③ AI 노드 에러 처리 정책** (Top3 선정, AI 요약)

- `onError: "continueRegularOutput"` → 실패해도 다음 노드 진행
- 또는 `onError: "stopWorkflow"` → 깨끗하게 중단

**④ 빈 결과 가드** (Merge 이후)

- Deduplicate News 코드에 `if (items.length === 0) return [];` 체크 추가
- 빈 배열이면 이후 AI 호출을 하지 않도록

**⑤ AI 요약의 재시도 상한** (이미 적용됨 ✅)

- `maxTries: 3`으로 설정 완료

## 4. RSS 피드가 없는 사이트를 강제로 크롤링하여 저작권 침해하지 않도록 설정

### **✅ RSS Feed 상태 확인 결과**

| **언론사** | **URL** | **상태** | **Content-Type** | **정상 여부** |
| --- | --- | --- | --- | --- |
| **연합뉴스 (YNA)** | `yna.co.kr/rss/news.xml` | `200 OK` | `application/xml` | ✅ 정상 |
| **한국경제** | `hankyung.com/feed/all-news` | `200 OK` | `text/xml` | ✅ 정상 |
| **파이낸셜뉴스** | `fnnews.com/rss/r20/fn_realnews_all.xml` | `301 → 200` | `application/xml` | ⚠️ 주의 |

### **⚠️ fnnews 주의사항**

파이낸셜뉴스는 **HTTP → HTTPS로 301 리다이렉트**가 발생합니다.

- 현재 워크플로우의 URL: `http://www.fnnews.com/...`
- 실제 응답 URL: `https://www.fnnews.com/...`

n8n의 HTTP Request 노드는 기본적으로 리다이렉트를 자동으로 따라가지만, **간헐적으로 실패할 수 있습니다.** 안전하게 URL을 `https`로 수정하는 것을 권장합니다. (수정완료)

### RSS 피드가 적용되지 않는 기사를 가져오게 될때

1. 우선적으로는 RSS피드가 없는 기사는 가져와서는 안되고,
2. RSS피드가 없는 기사를 가져오게 될 경우, 기사의 하단부에 "저작권 침해"와 유사한 내용의 경고문이 있다면 AI요약 전에 사용자에게 에러메시지로 먼저 알려주고, 기사의 제목만 가져오든지, 다른 기사를 가져오든지 두가지 액션을 선택할 수 있게

n8n은 기본적으로 **자동 실행** 워크플로우 도구라서, 실행 중 사용자에게 "A or B 선택하세요"를 보여주는 것이 쉽지 않습니다. 두 가지 접근 방식을 제안합니다:

|  | **옵션 A: 자동 분기 🤖** | **옵션 B: 실행 중단 + 수동 선택 ⏸️** |
| --- | --- | --- |
| 동작 | 저작권 경고 → 자동으로 제목만 사용 | 저작권 경고 → 워크플로우 일시정지 → 알림 전송 → 사용자 클릭으로 선택 |
| 장점 | 단순, 안정적, 완전 자동화 | 사용자가 직접 판단 가능 |
| 단점 | 사용자 개입 불가 | 구현 복잡, 이메일/슬랙 연동 필요 |

#### **핵심 차이 요약**

|  | **옵션 A (자동 분기)** | **옵션 B (수동 선택)** |
| --- | --- | --- |
| 추가 노드 | **4개** | **8~9개** |
| 사용자 개입 | 없음 (완전 자동) | 이메일/슬랙으로 선택 |
| "다른 기사" 교체 | ❌ 불가 | ✅ 가능 |
| 외부 서비스 필요 | ❌ | ✅ (이메일/슬랙 + 외부 Webhook) |
| localhost 환경 | ✅ 동작 | ⚠️ Webhook 제한 |

현재 **localhost Docker 환경**이기 때문에 옵션 B의 Wait 노드(Webhook)가 외부에서 접근되지 않아 제약이 있습니다.

### 새로 추가된 노드 (4개)

5. `YNA 오류 처리` — Code 노드 (신규)
- **위치**: YNA의 오류 출력(error output)에 연결
- **동작**:
  - `console.error()`로 n8n 실행 로그에 기록
  - `_rssError: true` 마커 포함 오류 정보 반환
- **연결**: → Merge Sources (input 0) + → RSS 오류 알림 (Notion)

6. `Hankyung 오류 처리` — Code 노드 (신규)
- **위치**: Hankyung의 오류 출력에 연결
- **동작**: 위와 동일 (소스명만 다름)
- **연결**: → Merge Sources (input 1) + → RSS 오류 알림 (Notion)

7. `fnnews 오류 처리` — Code 노드 (신규)
- **위치**: fnnews의 오류 출력에 연결
- **동작**: 위와 동일 (소스명만 다름)
- **연결**: → Merge Sources (input 2) + → RSS 오류 알림 (Notion)

8. `RSS 오류 알림 (Notion)` — Notion 노드 (신규)
- **자격증명**: 기존 `Notion OAuth2 API` 재사용
- **저장 DB**: `뉴스자동화` (기존과 동일)
- **저장 형식**:

| 필드 | 내용 |
|------|------|
| 제목 | ⚠️ [RSS 오류] 연합뉴스 (YNA) 수집 실패 |
| 요약문 | 3회 재시도 후에도 RSS 피드 수집 실패 (시각) |
| 발행일시 | 오류 발생 시각 (ISO 형식) |
| 언론사 | 시스템 오류 |

### 전체 오류 처리 흐름

```
RSS 피드 요청
    ↓ (실패 시 3초 간격으로 최대 3회 재시도)
    ↓ 3회 모두 실패
[YNA/Hankyung/fnnews] 오류 처리 노드
    ├──→ RSS 오류 알림 (Notion) 🔔  오류 내용 DB에 저장
    └──→ Merge Sources              _rssError 마커 전달
              ↓
         Deduplicate News           _rssError 항목 필터링 후 제거
              ↓
         나머지 정상 소스로 계속 진행 ✅
```

> [!NOTE]
> 3개 RSS 소스 중 **일부가 실패해도** 나머지 소스의 뉴스로 워크플로우가 정상 완료됩니다.  
> 실패한 소스는 Notion DB에 ⚠️ 표시로 기록되어 추적 가능합니다.
